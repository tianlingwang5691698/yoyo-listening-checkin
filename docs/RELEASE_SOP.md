# 提审前上线 SOP

## 目标

把当前工程收口到“可提审版本”，不把实验代码、调试残留和未验证流程带进上传包。

## 固定顺序

1. 整理工作区
2. 确认正式配置
3. 部署云函数
4. 真机自测
5. 生成真机稳定基线提交
6. 去掉开发态调试提示
7. 真机复测
8. 上传代码
9. 生成体验版
10. 准备提审材料
11. 发起提审

## 版本对应关系

每次发版前必须明确记录这 3 个版本点：

- Git 回退标签
- `cloudfunctions/yoyo/package.json` 版本
- 小程序上传说明里的版本说明

要求：

- Git 回退标签与云函数版本号保持一致
- 云函数部署后，CloudBase 中实际运行版本必须与当前标签对应
- 若页面代码和云函数不同步发布，必须在发布记录中明确标注

建议格式：

```text
Git tag: 3.0.1
Cloud Function: 3.0.1
Mini Program Upload Note: 3.0.1
```

- 首页必须是 `syncMode = cloud`

- 按真实能力填写，不多报权限
- 如歌曲目录为空，需说明该模块的展示逻辑

## 可提审最小能力

- 音频任务稳定播放
- 打卡与成长记录可用
- 家庭共享可用
- 家长日报可读
- 页面无明显调试残留
- 云端异常不会造成白屏

## 回退动作

出现严重问题时，按下面顺序回退：

1. 先确认问题发生在页面包还是云函数
2. 若是云函数问题，先回退 `yoyo` 到对应 tag 版本并重新部署
3. 若是页面问题，回退 Git tag 后重新上传小程序包
4. 回退后必须重新执行真机关键链路复测

标准回退命令：

```bash
git checkout <tag>
```

例如：

```bash
git checkout 3.0.1
```

说明：

- 只切 Git tag 不代表云函数自动回退，仍需重新部署 `cloudfunctions/yoyo`
- 只部署云函数不代表小程序页面自动回退，若页面也改过，需重新上传小程序包
