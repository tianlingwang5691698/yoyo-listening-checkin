# 设计风格与产品语言要求

本文件沉淀对话中已经明确的设计风格、交互要求和禁止项。每次进入计划模式、制定 UI 方案或实施 UI 改动前，必须先阅读本文、[PRODUCT_DESIGN_RULES.md](/Users/wangtianlong/工作/工作流/微信小程序/佑佑听力打卡/docs/PRODUCT_DESIGN_RULES.md) 和 [UI_CHANGE_CHECKLIST.md](/Users/wangtianlong/工作/工作流/微信小程序/佑佑听力打卡/docs/UI_CHANGE_CHECKLIST.md)。

## 目标气质

- 整体产品语言参考 Tesla 和 Apple：高级、极简、安静、克制。
- 不做报表，不做控制台，不做组件堆叠。
- 页面应像产品，而不是功能说明书。
- 少即是多：减少解释、减少按钮、减少状态、减少装饰。

## 首页要求

- 首页标题保持：`{{child.nickname || '佑佑'}}的英语小耳朵`。
- `英语小耳朵` 是产品识别的一部分，不应降级成副标题。
- 首页和“今日”要有关系，但不能把品牌标题改成“今日打卡”。
- 首页只负责进入学习，不承担数据看板职责。
- 不展示后台式节目计数，例如 `0/1`。
- 不展示独立总进度报表卡。
- Peppa / Unlock 1 / Songs 是主要入口，顺序固定。
- 如果节目卡或节目行整块可点击，就不要再写“开始”。
- 不允许同一任务同时有主按钮入口和列表入口。
- 推荐方向：品牌标题 + 横向节目卡；或者品牌标题 + 三条极简节目 cell。

## 课程页要求

- 课程页是播放体验，不是说明书。
- 主角应是音频播放和文本同步。
- 不展示普通用户不需要的技术链路：
  - `temp-url`
  - `ready`
  - `音频来源`
  - `播放链路`
  - `播放状态：已就绪`
- 不重复展示同一个进度，例如同屏多次出现 `1/3`、`0/3`、`已完成 0/3`。
- 三遍状态只能有一个主要表达位置。
- 播放键在真机和调试器中必须尽量一致；避免依赖原生 `button loading` 造成样式差异。
- 倍速不需要解释提醒，只需要在控制区显示 `×1.0` 或实际倍速值。
- 文本区不应写成“文本已就绪”式系统状态堆叠；优先让文本本身出现。
- 文本状态只在没有文本或准备中时短句提示。

## 文案要求

- Songs 始终显示英文 `Songs`，页面层不出现“歌曲”。
- 禁止用大写英文 eyebrow 制造高级感，例如 `PROGRAM CONSOLE`。
- 禁止解释型长句堆叠。
- 卡片说明不超过 28 个中文字。
- 状态文案要短：
  - `完成`
  - `等待`
  - `进行中`
  - `可追赶`
  - `无需追赶`
- 不用“功能说明”代替产品语言。

## 字体要求

- 全项目遵循 6 个层级：
  - `type-brand`
  - `type-page-title`
  - `type-card-title`
  - `type-section-title`
  - `type-body`
  - `type-caption`
- 普通标题不使用 `font-weight: 900`。
- `900` 只允许用于成长页大数字或统计数字。
- 不使用负字距。
- 不使用夸张 `letter-spacing`。

## 视觉要求

- 背景使用暖白或近白。
- 主强调色只留给真正主操作或关键状态。
- 底部 tab 选中态使用深灰/近黑，不使用主操作橙。
- 卡片要轻，阴影要浅。
- 禁止装饰性渐变、光斑、复杂背景。
- 同一页面不超过 2 种卡片权重。

## 交互要求

- 整行可点击时，不再额外显示“开始”。
- 待准备任务不可跳转。
- 多任务子行必须使用阻止冒泡的点击方式。
- 首页节目卡点击进入当前节目任务。
- 课程页播放、回退、倍速、结束计数、transcript 高亮是核心链路，UI 改动不能破坏。

## 计划与实施纪律

- 每次计划模式前必须先阅读本文。
- 每次 UI 实施前必须先阅读本文和检查清单。
- 每次改动后必须跑回归检查。
- 检查通过后再提交。
- 最终回复必须包含下一步建议。
