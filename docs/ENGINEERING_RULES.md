# 项目工程演进规则

## 目标

本项目的目标不是做“看起来复杂”的架构，而是保持：

- 业务规则清楚
- 页面层简单
- 云端逻辑可维护
- 后续继续加功能时不容易变乱

后续所有开发、重构、修复，都优先遵守下面规则。

## 1. 页面层只做展示和交互

`pages/*` 只负责：

- 展示数据
- 响应点击
- 触发跳转
- 调用 `utils/store`
- 处理轻量 UI 状态

`pages/*` 不负责：

- 任务推进规则
- 打卡规则
- 补卡规则
- 日报统计规则
- 计划生成规则

如果一个逻辑会影响“业务结果”，不要写在页面里。

## 2. store 只做统一数据入口

`utils/store.js` 只负责：

- 调云函数
- 提供默认返回结构
- 包装错误态
- 统一前端调用方式

`utils/store.js` 不负责：

- 业务规则判断
- 状态推进
- 日期推进
- 补卡口径
- 任务完成逻辑

store 是“入口层”，不是“规则层”。

## 3. service 只做请求编排

`cloudfunctions/yoyo/services/*` 只负责：

- 接收 action 请求
- 读取上下文
- 组织调用顺序
- 调用 shared facade / engine / repository
- 返回结果

`services/*` 不应该长期堆积复杂规则。

如果某段逻辑满足下面任一条件，应优先拆到 engine：

- 有明确业务规则
- 会被多处复用
- 以后大概率继续变
- 改动后容易影响多个页面或 action

## 4. 新业务规则优先进 engine

以下逻辑默认属于 engine：

- 中国时间日期规则
- planDayIndex 推进
- 当天任务生成
- 打卡完成判定
- 补卡起点与漏卡判定
- 日报生成口径
- heatmap / catchup 数据计算
- A1/A2/B1/B2 任务装配
- 课程详情规则拼装

简单说：

- 有“判断规则”的，优先进 engine
- 有“只是调库”的，进 repository
- 有“只是接请求”的，进 service

## 5. repository 不写业务判断

`repositories/*` 只负责：

- 查
- 增
- 改
- upsert
- 删除
- 封装数据库访问细节

`repositories/*` 禁止写：

- 能不能补卡
- 今天是否已完成
- 是否应该推进到下一天
- 哪一天该显示哪批任务
- 哪天算漏卡

repository 只管“拿数据 / 写数据”，不决定“业务意义”。

## 6. shared facade 不再扩张

`shared.service.js` 的定位是：

- 聚合公共依赖
- 暴露跨 service 的公共能力
- 提供基础 facade
- 做少量上下文准备

`shared.service.js` 不再新增大块复杂业务规则。

如果新增逻辑超过下列任一情况，就不要放进 shared：

- 超过 20 行且含多处判断
- 可被多个 service 复用
- 有明显业务语义
- 未来会单独演进

优先新建独立 engine。

## 7. 新返回结构先定契约

新增或修改以下返回数据前，必须先想清楚字段契约：

- DashboardData
- ReportData
- CatchupState
- TaskDetail
- LevelOverview

至少明确：

- 字段名
- 字段类型
- 字段含义
- 是否必填
- 默认值策略

目标是：

- 不靠“脑补字段”
- 不让页面猜返回结构
- 不让后续改动反复破页面

## 8. 核心规则改动必须补测试

改下面这些逻辑时，必须有对应测试或最少可复验脚本：

- 中国零点切换
- planDayIndex 推进
- 当天完成后不提前跳次日
- 补卡起点
- catchupState
- 完成打卡
- 日报生成
- 最近 7 天 / heatmap 数据

没有测试保护时，不做大规模规则重构。

## 9. 日期统一走中国时区工具

所有“今天”“次日”“日期截断”“零点刷新”相关逻辑，统一走中国时区日期工具。

禁止继续散写：

```js
new Date().toISOString().slice(0, 10)
```

统一使用项目内日期能力，例如：

- `china-date`
- `getTodayString`
- `formatChinaDateFromDate`
- `addDays`
- `diffDays`

原则：

- 业务日期一律按中国自然日
- 存储时间可以保留 ISO 时间戳
- 业务判断不要直接拿 UTC 日期切片

## 10. 改动前先判断归属层

每次改功能前，先判断这段逻辑属于哪一层：

- 页面展示：`pages/*`
- 数据入口：`utils/store.js`
- 请求编排：`services/*`
- 业务规则：`lib/*-engine` 或 `lib/*`
- 数据存取：`repositories/*`
- 外部依赖适配：`adapters/*`

如果归属层判断不清，先不要写代码。

先找对位置，再实现。

## 11. 优先做“收口”，不要做“绕路”

新增功能时，优先把逻辑收进现有规则层，而不是临时在页面或 service 里绕开。

允许：

- 在现有 engine 中扩展规则
- 新增独立 engine
- 在 facade 中补最少接线

不鼓励：

- 页面里临时判断
- service 里复制旧规则
- repository 里偷塞业务逻辑
- 为赶进度先写一套旁路逻辑

临时方案如果进入主分支，最后通常都会变成长期负担。

## 12. 重构优先顺序

以后继续优化架构，优先顺序固定为：

1. 先立规则
2. 再补测试
3. 再固化契约
4. 再拆 shared / service
5. 最后再做领域彻底切分

不要一上来就大拆目录。

先保证规则稳定，再做结构漂亮。

## 13. 当前推荐的工程层次

当前项目建议长期维持为：

- `pages/*`：展示层
- `utils/store.js`：前端入口层
- `services/*`：云函数编排层
- `shared.service.js`：公共 facade
- `lib/*-engine.js`：业务规则层
- `lib/*`：基础运行时逻辑
- `repositories/*`：数据访问层
- `adapters/*`：外部环境适配层

所有新增逻辑优先往这个结构里归位，不要重新发明平行层。

## 14. 最终判断标准

判断一个改动是不是“工程上正确”，优先看这 4 件事：

- 规则是不是放在正确层
- 改动后是否更容易测试
- 返回结构是否更稳定
- 后续同类需求是否更容易接着做

如果只是“眼前能跑”，但会让结构更乱，那不算正确方案。
